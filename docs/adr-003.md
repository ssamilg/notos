# ADR-003: Feature Addendum 1 Implementation Strategy

**Date:** 2026-06-26
**Status:** Accepted

## Context
The product requirements have expanded to include Markdown rendering, task state management ("completed"), global tag management, search/filtering, and various UX enhancements. Previous design iterations favored in-place editing; however, Markdown rendering requires a distinct separation between raw input and formatted output. Furthermore, manual sorting across offline clients introduces extreme synchronization complexity (fractional indexing).

## Decision
1. **Markdown & Interaction:** Revert to an explicit "View / Edit" toggle model for notes. The View mode will strictly render Markdown, while the Edit mode will expose the raw text area.
2. **Tag Architecture:** Deprecate the flat `tag` column on notes. Introduce a normalized schema with `tags` and `note_tags` (many-to-many) to support global tag management and safe deletions.
3. **Sorting:** Defer manual drag-and-drop sorting to a future phase. Implement a default sort order where `is_completed = true` notes are routed to the bottom of the list.
4. **Pagination:** Implement cursor-based infinite scrolling on the API and UI layers instead of traditional offset pagination.

## Consequences

### Positive
- Normalizing tags prevents data anomalies and supports global renaming/deletion easily.
- Cursor-based infinite scroll is highly performant for large datasets.
- Deferring manual sorting allows the team to deliver high-value features (Markdown, Search) without getting bogged down in complex synchronization algorithms.

### Negative
- Reintroducing the View/Edit toggle adds slight navigational friction compared to pure in-place editing, but is necessary for proper Markdown support.
- Many-to-many tag relationships require more complex backend service logic and database transactions.
