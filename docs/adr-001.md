# ADR-001: Core Infrastructure & Offline Sync Strategy

**Date:** 2026-06-23
**Status:** Accepted

## Context
NOTOS requires a latency-free, offline-first experience with optimistic UI rendering[cite: 1]. Furthermore, the web application must serve as a foundation for a future React Native mobile port. We needed to define the API routing layer, the local persistence strategy, and the execution environment for background cleanup jobs.

## Decision
1. **API Layer:** We will use **Next.js Route Handlers (`app/api`)** instead of a standalone Express server to align with the Next.js framework[cite: 5]. 
2. **Offline Persistence:** We will use **IndexedDB** (via a lightweight wrapper like `localForage`) connected to React Context[cite: 5].
3. **Background Jobs:** We will use the native **Supabase `pg_cron` extension** to handle soft-delete flushes[cite: 1, 5].

## Consequences

### Positive
- The Next.js API layer allows the future mobile app to consume the exact same endpoints without rewriting backend infrastructure.
- IndexedDB handles large JSON payloads asynchronously, ensuring the UI thread remains unblocked and maintaining the "zero perceived latency" requirement[cite: 1].
- `pg_cron` keeps database-lifecycle logic natively inside the database, bypassing serverless HTTP timeouts and security exposure.

### Negative
- IndexedDB introduces slight complexity compared to `localStorage`, requiring an asynchronous wrapper.
- `pg_cron` requires writing raw SQL for the cron definition rather than using JavaScript/TypeScript.
