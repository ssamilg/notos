# ADR-005: UI/UX Standardization & Interaction Refinements

**Date:** 2026-06-27
**Status:** Accepted

## Context
As the application matures, the interaction models need to be standardized to prevent accidental data loss and optimize the "pure signal" capture workflow. The previous "in-place editing" pattern for list items proved too fragile for destructive actions (deletions). Furthermore, users need faster, repetitive capture mechanics (Save & New) and cleaner metadata visualization.

## Decision
1. **List Layouts:** Deprecate pure "click-to-edit" for Project and Tag lists. Implement a standardized, explicit layout: Row 1 (`NAME | EDIT BTN`), Row 2 (`STATS | DATES`). Clicking "Edit" transitions the row into a mutation state (`SAVE | CANCEL | DELETE`).
2. **Data Safety:** Enforce a universal confirmation modal for all destructive operations (Project, Note, and Tag deletions).
3. **Capture Workflow:** Implement a "Split-Button" for saving notes with three discrete actions: `Save`, `Save & Exit`, and `Save & New Note`. The user's last selected action will be persisted to `localStorage` and become the default primary action for future captures.
4. **Metadata & Tags:** Standardize dates to relative formats (falling back to absolute after 3 months) with absolute hover tooltips. Format tags natively with brackets and hashes (`[#tag1, #tag2]`).
5. **Tag Input:** Utilize a global autocomplete pool for tag suggestions. Tag creation triggers on `Enter`; tag removal triggers on `Backspace` when the input is empty.

## Consequences

### Positive
- Explicit "Edit" modes and universal delete modals virtually eliminate accidental data destruction.
- The persistent Split-Button dramatically speeds up repetitive note-taking workflows tailored to individual user preferences.
- Relative dates and bracketed tags reinforce the premium, terminal-like minimalist aesthetic.

### Negative
- Component complexity increases for list items (managing `isEditing` mode and toggling action button groups).
- `localStorage` introduces a minor secondary state layer outside of React Query that must be managed for the Save button preference.
