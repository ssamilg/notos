# ADR-006: Monorepo Architecture & Client-Server Split

**Date:** 2026-07-06
**Status:** Accepted

## Context
The application initially utilized a Next.js full-stack approach. However, for a strictly offline-first SPA that does not rely on Server-Side Rendering (SSR) or SEO, Next.js introduced unnecessary overhead and a suboptimal developer experience. To ensure strict boundary separation and improve local development performance, the architecture must transition to a decoupled client-server model.

## Decision
1. **Repository Structure:** Adopt **Turborepo** to manage a monorepo containing distinct frontend, backend, and shared packages.
2. **Frontend (`apps/web`):** Migrate the Next.js frontend to a **React + Vite** SPA architecture to maximize dev server speed and strip out server-routing overhead.
3. **Backend (`apps/api`):** Migrate Next.js Route Handlers to a standalone **Node.js + Express** server.
4. **Shared Types (`packages/shared`):** Create a dedicated workspace to house shared TypeScript interfaces, Zod validation schemas, and database type definitions.
5. **Authentication:** Implement a custom Express middleware to validate Supabase JWTs via standard `Authorization: Bearer <token>` headers.
6. **Network Boundary:** Configure Express CORS to strictly whitelist the local Vite development port (`http://localhost:5173`) during development.

## Consequences

### Positive
- Strict separation of concerns between presentation and backend infrastructure.
- The `packages/shared` workspace guarantees that frontend API calls and backend route validations never fall out of sync.
- Vite offers a significantly faster Hot Module Replacement (HMR) experience for SPA development.

### Negative
- Requires initial DevOps overhead to configure Turborepo pipelines, linting, and TS configuration across multiple workspaces.
- Loss of Next.js edge middleware requires re-implementing JWT validation logic in Express.
