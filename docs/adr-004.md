# ADR-004: State Management Rewrite & Cache Simplification

**Date:** 2026-06-26
**Status:** Accepted

## Context
The application's development phase began suffering from boilerplate bloat due to the manual orchestration of offline state (React Context + custom IndexedDB synchronization queues). While the offline-first requirement remains, writing bespoke code to manage temporary ID resolution, optimistic rollbacks, and sync queues introduced unnecessary complexity and friction for a simple note-taking application.

## Decision
1. **State Management:** Deprecate custom React Context and manual IndexedDB wrappers. Adopt **React Query (TanStack)** to natively handle remote state caching, background refetching, and offline mutation queues.
2. **Primary Keys:** Shift to **Client-Generated UUIDs** (`crypto.randomUUID()`). The frontend will generate the final ID upon creation to enable instant relational linking and optimistic rendering without waiting for a server roundtrip.
3. **Cache Lifecycle:** Simplify the persistence model. A hard page refresh will clear the local memory cache and trigger a fresh fetch from the Supabase database via the Next.js API. IndexedDB will only be used by React Query as an ephemeral backup if offline routing is needed during an active session.

## Consequences

### Positive
- Massively reduces frontend codebase size by deleting manual state orchestrators and sync queues.
- Client-generated UUIDs eliminate the need for "temporary ID" mapping scripts and prevent relational data collisions when creating entities offline.
- React Query natively handles loading states, error states, and optimistic UI rollbacks (`onMutate` / `onError`), standardizing component architecture.

### Negative
- The API layer must be refactored to trust and accept client-provided primary keys.
- Requires a one-time intensive refactor to rip out the existing Context providers across all components.
